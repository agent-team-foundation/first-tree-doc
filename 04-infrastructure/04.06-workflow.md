# Workflow

Not everything should be handled by open-ended agent reasoning. Some processes are deterministic. If A then B, always. Workflow infrastructure handles these cases.

## Why it exists alongside agents

```
┌──────────────────────┐     ┌──────────────────────┐
│   Agent reasoning    │     │   Workflow            │
│                      │     │                       │
│   ambiguous tasks    │     │   deterministic tasks │
│   judgment calls     │     │   if A then B         │
│   interpretation     │     │   routing approvals   │
│   design decisions   │     │   triggering deploys  │
│                      │     │   sending notifs      │
│   flexible           │     │   reliable            │
│   unpredictable      │     │   auditable           │
└──────────────────────┘     └───────────────────────┘

Use agents for judgment.
Use workflows for correctness.
```

Mixing deterministic processes into an agent loop adds unnecessary complexity. Route an approval? Workflow. Decide whether to approve? Agent.

## How it works

- Workflows are defined as code, not graphical editors. Structured scripts any agent can read and invoke.
- Triggered by events: a PR merge, a message received, a scheduled time. Analogous to GitHub Actions.
- The workflow layer exposes a CLI. Agents invoke workflows by name with parameters.
- Workflows can call agents. Agents can trigger workflows. The two layers compose.

## Design principle

Keep the rule set small. Every workflow is a rule that must be maintained. Prefer agent judgment for anything that requires interpretation. Use workflows only where correctness matters more than flexibility.
