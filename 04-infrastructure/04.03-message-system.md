# Message System

When an agent needs something from a domain it does not own, it cannot just read the tree and act. It needs to reach the owner. The message system is how that happens.

## How it works

```
Agent A                           Agent B
(owns /backend/)                  (owns /frontend/)

  "I need to change the     ──▶  inbox receives message
   API contract. Does the
   frontend need updating?"       spawns new instance
                                  loads tree context
                             ◀──  "yes, the dashboard
                                   component depends on
                                   that endpoint"

                             ──▶  both update their nodes
                                  tree records the decision
```

1. Agent sends a message. Recipient's inbox receives it.
2. Recipient spawns a new instance to handle it. Main thread stays unblocked.
3. Spawned instance loads context from the tree, processes the request, replies.
4. The exchange may produce a tree update.

## Group communication

Agents and humans can share a group channel.

- An agent participates only when addressed directly. This prevents noise and uncontrolled token consumption.
- Each message exchange is a discrete session with fresh context.

## Agent-to-agent routing

Most messaging platforms (Slack, Discord, Feishu) do not support bot-to-bot communication natively. The message system provides a server-side routing layer so agents can message each other.

## Protocol

- Open. Any agent that can send and receive structured messages can participate.
- Composes with existing platforms (Slack, Discord, Feishu, GitHub notifications).
- The goal is to work with what teams already use, not replace it.
