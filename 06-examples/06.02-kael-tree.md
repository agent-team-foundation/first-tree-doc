## What it demonstrates

### Organized by concern, not by repo

Kael is four repos (backend, frontend, extension, desktop). But the tree is organized by what an agent needs to know: `agent/`, `knowledge/`, `chat/`, `environment/`, `platform/`. An agent working on auth goes to `/platform/auth/` and finds everything in one place.

### AGENT.md as the entry point

The root has an `AGENT.md` that instructs agents: read the tree before every task, design in the tree, implement in code, ask "does the tree need updating?" after every task.

### Design decisions with rationale

Leaf nodes capture WHY, not just WHAT. The memory system node explains why MemGPT was chosen over four alternatives, with a comparison table. An agent reading this can make informed decisions.

### Design docs split between tree and code

The tree captures the decision and cross-repo implications. Code repos keep the implementation plan. They link to each other. This is the decision/execution layer separation in practice.

### The design/execute test

From principles.md: "If an agent needs this to _design_ a task, it belongs in the tree. If only to _execute_, it stays in the source system." This keeps the tree lean and high-signal.

### Draft first, restructure later

From practices.md: start by writing what you know. Restructure when scanning gets noisy. The structure follows the content, not the other way around.
