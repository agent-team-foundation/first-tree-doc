# Quick Start

Create your first Context Tree in five minutes.

## Step 1: Create a repo

```bash
mkdir my-team-tree
cd my-team-tree
git init
```

## Step 2: Create the root node

Every tree starts with a `NODE.md` at the root.

```bash
cat > NODE.md << 'EOF'
---
title: "My Team"
owners: [your-github-username]
---

# My Team

This is the root of our Context Tree.
EOF
```

This is the top of your tree. It describes your organization and declares who owns the root.

## Step 3: Add your first subtrees

Create folders for your team's domains. Every folder needs a `NODE.md`.

```bash
mkdir -p product engineering members

cat > product/NODE.md << 'EOF'
---
title: "Product"
owners: [product-lead]
---

# Product

Product decisions, roadmap, and design docs.
EOF

cat > engineering/NODE.md << 'EOF'
---
title: "Engineering"
owners: [eng-lead]
---

# Engineering

Architecture decisions, technical designs, and infrastructure.
EOF

cat > members/NODE.md << 'EOF'
---
title: "Members"
owners: ["*"]
---

# Members

Team members: humans and agents.
EOF
```

Your tree now looks like this:

```
my-team-tree/
├── NODE.md              ← root
├── product/
│   └── NODE.md          ← product domain
├── engineering/
│   └── NODE.md          ← engineering domain
└── members/
    └── NODE.md          ← team members
```

## Step 4: Add yourself

Create a personal node under `members/`.

```bash
cat > members/alice.md << 'EOF'
---
title: "alice"
owners: [alice]
role: "Engineer"
domains:
  - "backend"
  - "infrastructure"
---

## About

Backend engineer. Focused on API design.

## Current Focus

- Designing auth system v2
EOF
```

## Step 5: Add your first decision

Write a leaf node that records a real decision.

```bash
cat > engineering/why-monolith.md << 'EOF'
---
title: "Why Monolith"
owners: [eng-lead]
status: approved
---

# Why Monolith

We chose a monolith over microservices because:
- Team is 4 people. Microservices add operational overhead we can't afford.
- We can always split later. Merging back is harder.
- Deployment is one thing, not twelve.

Decision made 2025-01-15.
EOF
```

This is what makes a Context Tree different from a wiki. The node records the decision AND the reasoning.

## Step 6: Add an AGENT.md

Create an `AGENT.md` at the root. This is the instruction file that agents read before every task.

```bash
cat > AGENT.md << 'EOF'
# Agent Instructions

## Before Every Task

1. Read the root NODE.md to understand the domain map.
2. Read the NODE.md of every domain relevant to your task.
3. Follow soft_links. If a node declares them, read those nodes too.
4. Read leaf nodes that match your task.

## During the Task

- Design in the tree, implement in code.
- The tree captures the WHY. Code captures the HOW.
- Respect ownership. If your changes touch a domain you don't own, flag it.

## After Every Task

Ask: does the tree need updating?
- Did you discover something the tree didn't capture?
- Did you find the tree was wrong or outdated? Fix it.
EOF
```

This is the equivalent of a `CLAUDE.md` but for the tree. Any agent that works with your tree reads this first.

## Step 7: Push it

```bash
git add -A
git commit -m "init tree"
git remote add origin https://github.com/your-org/your-tree.git
git push -u origin main
```

## What you have now

```
my-team-tree/
├── NODE.md
├── product/
│   └── NODE.md
├── engineering/
│   ├── NODE.md
│   └── why-monolith.md      ← your first decision
└── members/
    ├── NODE.md
    └── alice.md              ← your first member
```

A tree with three domains, one member, and one decision. It is small. It will grow.
