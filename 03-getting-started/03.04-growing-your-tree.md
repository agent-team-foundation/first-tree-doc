# Growing Your Tree

A tree starts small. Three folders, a few nodes. Over time it grows. This page covers when and how to restructure.

## Draft first, restructure later

Do not wait for the perfect structure before writing. An imperfect node is more useful than a missing one.

Start by writing what you know. Append knowledge as you discover it. When a node accumulates so much content that an agent can no longer scan it quickly, that is the signal to restructure: split into subdomains, consolidate redundant entries, or rewrite for clarity.

The structure should follow the content, not precede it.

## When to split

Split a folder when it gets too broad.

```
Before:
  /engineering/
    NODE.md
    why-monolith.md
    why-postgres.md
    api-design.md
    auth-v2.md
    deployment-strategy.md
    ci-pipeline.md

Too many topics in one folder. An agent arriving here
does not know where to look.

After:
  /engineering/
    NODE.md
    architecture/
      NODE.md
      why-monolith.md
      api-design.md
    backend/
      NODE.md
      why-postgres.md
      auth-v2.md
    infrastructure/
      NODE.md
      deployment-strategy.md
      ci-pipeline.md

Each subfolder is a clear domain with its own owner.
```

A good rule: if you have more than 7-10 leaf nodes in a folder, consider splitting.

## When to merge

Merge when two folders are so tightly coupled that they always need each other.

```
Before:
  /auth/
    NODE.md
    jwt-design.md
  /permissions/
    NODE.md
    rbac-model.md

These are always read together.
Every decision in one affects the other.

After:
  /auth/
    NODE.md
    jwt-design.md
    rbac-model.md

One domain. One owner. No ambiguity.
```

## How to deprecate

Node owners deprecate nodes. Not agents. Not automation.

When a decision is replaced, the owner creates a new node and marks the old one as deprecated.

```yaml
# /engineering/auth-v1.md
---
title: "Auth V1 (Deprecated)"
owners: [alice]
status: deprecated
---

Deprecated. Replaced by auth-v2.md.
See /engineering/auth-v2.md for current design.
```

Deprecated nodes stay in the tree. They are not deleted. History matters. Someone might need to know why auth-v1 was replaced.

## Tree restructuring

When a new NODE.md is proposed, the tree structure should be optimized for agent navigation. Think about how an agent searches for knowledge:

```
Agent looking for "database decision":

  1. Start at root
  2. Walk down to /engineering/
  3. Walk down to /engineering/backend/
  4. Find /engineering/backend/why-postgres.md

The path should be intuitive.
If an agent cannot find a node by walking the tree,
the tree structure needs work.
```

## The decision layer stays clean

Remember: the tree is the decision layer. Code is the execution layer. They are separated.

When you restructure the tree, you are reorganizing decisions. You are not moving code. The tree lives in its own repo. Restructuring the tree does not affect any codebase.

This makes restructuring safe. You cannot break production by reorganizing your decisions.
