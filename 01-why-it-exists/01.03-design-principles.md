# Design Principles

Context Tree is built on four core principles. They are deliberate constraints. Each one closes off complexity in exchange for clarity.

## Transparency by default

```
┌────────────────────────────────────┐
│         CONTEXT TREE               │
│                                    │
│   Reading:   open to everyone      │
│   Writing:   requires approval     │
│                                    │
│   Any agent can read any node.     │
│   Only owners can approve changes. │
└────────────────────────────────────┘
```

All information in the tree is readable by everyone: humans and agents alike. Writing requires owner approval. Reading is open.

An agent that cannot read the tree cannot build context from it. The whole point of shared memory is that it is shared.

## Agents as first-class participants

The tree is designed to be navigated by agents, not just humans.

```
Agent workflow:

  1. Start at home node
  2. Walk up the tree for broader context
  3. Read NODE.md files to understand domains
  4. Follow soft links for related context
  5. Act, with full team context
  6. Write back: new decision → new node
```

This means the format must be agent-friendly:

- Plain text, not proprietary formats
- Predictable structure, not freeform
- Machine-readable frontmatter (YAML)
- File paths as addresses, not opaque IDs

Every design decision asks: can an agent navigate this without special tooling?

## Tree over graph

```
Tree:                          Graph:

     root                      A ── B ── C
    /    \                     │ ╲  │ ╱  │
   A      B                   D ── E ── F
  / \    / \                   │ ╱  │ ╲  │
 C   D  E   F                 G ── H ── I

 Every node has one parent.    Everything connects to
 Walk up for broader context.  everything. Where do
 Walk down for detail.         you start? Where do
 Clear location.               you go?
```

A tree keeps information organized and navigable. Every node has exactly one parent. You always know where you are.

Graphs are more powerful in theory. In practice, they become unnavigable. When everything links to everything, nothing has a clear location. Search becomes the only interface.

A tree forces you to decide where knowledge belongs. That constraint is the point.

Where cross-references are needed, [soft links](../02-how-it-works/02.05-soft-links.md) allow them without collapsing the tree into a graph.

## Git as the foundation

```
┌─────────────────────────────────────────────┐
│  Why git?                                    │
│                                              │
│  Every change is a commit         → history  │
│  Pull requests work out of the box→ review   │
│  CODEOWNERS from the tree         → enforce  │
│  Blame shows who wrote what       → trace    │
│  No proprietary database          → freedom  │
│  Works with existing tools        → adopt    │
└─────────────────────────────────────────────┘
```

Each node is a file. The tree is a repository. History, merges, and ownership follow the same model software engineering has refined for decades.

Git is not just the storage layer. It is the collaboration layer. The tree inherits decades of tooling for versioning, branching, merging, and reviewing changes to text files.

No vendor lock-in. The tree is markdown files in a git repo. You can host it anywhere.

The following principles are derived from [kael-tree](https://github.com/agent-team-foundation/kael-tree), a production Context Tree that extends the core four with operational guidance.

## Decision layer, not execution layer

The Context Tree is the _decision layer_. Codebases are the _execution layer_. They are separated.

```
┌──────────────────────┐     ┌──────────────────────┐
│   DECISION LAYER     │     │   EXECUTION LAYER    │
│   (Context Tree)     │     │   (Codebase)         │
│                      │     │                      │
│   why                │     │   how                │
│   what               │────▶│   code               │
│   who                │     │   configs             │
│   design decisions   │     │   implementations    │
│                      │     │                      │
│   tree repo          │     │   other repos        │
└──────────────────────┘     └──────────────────────┘

  Tree tells code WHAT to do.
  Code does NOT change what the tree decided.
  They live in separate repos.
```

**The test:** If an agent needs this information to _design_ a task, it belongs in the tree. If the agent only needs it to _execute_ the task, it stays in the source system.

```
Belongs in the tree:                Does NOT belong in the tree:

"Auth spans 4 repos: backend        "The function signature of
 issues JWTs, frontend uses          retrieval_service.search()"
 Better Auth, extension uses         → read the code
 OAuth popup, desktop uses
 localhost callback."                "The database schema for
→ agent needs this to design         chunk_embeddings table"
                                     → read the models
"We chose PostgreSQL because
 our data is relational and          "Current list of API endpoints"
 we need ACID for billing."          → read the route files
→ strategic decision with
  rationale                          "Stable, well-known patterns
                                      like 'we use FastAPI'"
"Ingestion pipeline: download         → everyone knows
 → extract → parse → chunk
 → embed → store"
→ agent would need to trace
  6+ files to piece this together
```

A node earns its place in the tree by compressing expensive knowledge: cross-domain relationships, strategic decisions, and constraints that would require an agent to traverse multiple source systems to derive.

Agents enforce the separation. A reviewer agent checks every PR against the tree. If code conflicts with a decision in the tree, the PR is blocked. A maintainer agent periodically checks alignment and flags drift.

The tree is the authority. Code follows. Not the other way around.

## Organize by concern, not by team

Domains in the tree are organized by _what an agent needs to know to act_, not by repo, team, or org chart.

```
Bad (organized by repo):        Good (organized by concern):

/kael-backend/                   /agent/
/kael-frontend/                    memory, tools, runtime
/kael-extension/                 /knowledge/
/kael-desktop/                     ingestion, retrieval, search
                                 /chat/
An agent working on "add SSO"      sessions, streaming, UI
has to search across all four    /environment/
repos to find auth context.        browser, desktop, sandbox
                                 /platform/
                                   auth, security, integrations

                                 An agent working on "add SSO"
                                 goes to /platform/auth/ and
                                 finds everything in one place.
```

A feature lives in the domain that owns the primary decision, with soft links to other domains for discoverability.
