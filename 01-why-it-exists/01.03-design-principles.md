# Design Principles

Context Tree is built on four principles. They are deliberate constraints. Each one closes off a class of complexity in exchange for clarity.

## Transparency by default

All information in the tree is readable by everyone: humans and agents alike. Writing requires owner approval. Reading is open.

This is a choice. Many knowledge systems default to restricted access and require explicit grants for reading. Context Tree inverts this. The whole point of a shared memory is that it is shared. An agent that cannot read the tree cannot build context from it.

Permissions and access controls will become more important as teams scale. But transparency is the starting point, not the exception.

## Agents as first-class participants

The tree is designed to be navigated and updated by agents, not just humans.

An agent starts from its assigned node and traverses the tree to build context before acting. It reads NODE.md files to understand domains. It follows soft links to find related context. It writes back when it completes work or makes a decision.

This means the tree format must be agent-friendly: plain text, predictable structure, machine-readable frontmatter. Markdown and YAML, not proprietary formats. File paths as addresses, not opaque IDs.

Every design decision in Context Tree asks the question: can an agent navigate this without special tooling?

## Tree over graph

A tree structure keeps information organized and navigable. Every node has exactly one parent. You can always walk up to get broader context, or walk down to get more specific.

Graphs are more powerful in theory. In practice, they become unnavigable. When everything can link to everything, nothing has a clear location. Search becomes the only interface, and search requires knowing what to ask.

A tree enforces hierarchy. This is a constraint, and it is the point. It forces teams to think about where knowledge belongs. Disagreements about structure are resolved by splitting into subtrees, not by adding more links.

Where cross-references are genuinely needed, [soft links](../02-how-it-works/02.05-soft-links.md) allow them without collapsing the tree into a graph.

## Git as the foundation

Each node is a file. The tree is a repository. History, merges, and ownership follow the same model that software engineering has refined for decades.

This means:

- Every change is a commit. You can see who wrote what and when.
- Pull requests and code review work out of the box. An agent proposing a change to a node it does not own creates a PR, and the owner reviews it.
- `CODEOWNERS` can be generated from the tree's ownership declarations, giving teams PR-level enforcement for free.
- No proprietary database. No vendor lock-in. The tree is files in a git repository. You can host it anywhere.
- The tree composes with existing developer workflows. It does not require teams to adopt new tools.

Git is not just the storage layer. It is the collaboration layer. The tree inherits decades of tooling for versioning, branching, merging, and reviewing changes to text files.
