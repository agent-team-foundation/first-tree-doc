# Ownership

Every node in the tree has an owner. This is how accountability works.

## The six rules

### Rule 1: NODE.md is the folder-level authority

Its owners can approve changes to any file within the folder.

```
/backend/
  NODE.md    owners: [alice]     ← alice controls everything here
  auth.md
  storage.md
```

### Rule 2: Leaf file owners are additive

A leaf file can declare its own owners. These are added to the NODE.md owners, not replacing them.

```
/backend/
  NODE.md      owners: [alice]
  auth.md      owners: [carol]

Who can approve changes to auth.md?
→ Alice (from NODE.md) AND Carol (from auth.md)
```

### Rule 3: Multiple owners are allowed

On both NODE.md and leaf files.

```yaml
owners: [alice, bob, carol]
```

### Rule 4: Inheritance with override

If a NODE.md does not declare owners, or declares `owners: []`, it inherits from the nearest parent that does.

When a NODE.md explicitly declares owners, it fully overrides the parent. The parent's owners have no implicit authority over child folders.

```
/                    owners: [founder]
├── backend/         owners: [alice]       ← overrides founder
│   └── auth.md                            ← alice owns this
└── docs/            owners: []            ← inherits from root
    └── readme.md                          ← founder owns this

owners: []    = inherit from parent
owners: [bob] = bob owns this, overrides parent
```

### Rule 5: Every folder requires a NODE.md

Creating a subfolder means someone has to think about ownership. Either declare owners explicitly or inherit from the parent.

No NODE.md, no folder.

### Rule 6: Wildcard owner

`owners: [*]` means anyone can approve changes. The parent NODE.md owners still retain authority.

```
/backend/
  NODE.md      owners: [alice]
  tips.md      owners: [*]

Who can approve changes to tips.md?
→ Anyone. Alice also retains authority.

Good for: community content, FAQs, contribution guides.
```

> **Note:** GitHub's YAML parser requires the wildcard to be quoted: `owners: ["*"]`. The spec uses `owners: [*]`. Both mean the same thing.

## Conflict resolution

When a node has multiple owners and a PR is submitted, **all owners must approve**. If any owner rejects, the change is blocked.

```
/backend/auth.md    owners: [alice, carol]

  Agent submits PR:

  alice approves ──┐
                   ├──▶ both approve → merged
  carol approves ──┘

  alice approves ──┐
                   ├──▶ carol rejects → blocked
  carol rejects  ──┘
```

If owners keep disagreeing, split the domain into two subtrees with clear single owners.

## Fallback

Mistakes happen. The fallback is always a new PR.

Merged something wrong? Open a new PR to revert. The revert is recorded in git history. Clean history. No out-of-band changes.

## Summary

The six rules (from the [Context Tree spec](https://github.com/agent-team-foundation/first-tree)):

```
┌───────────────────────────────────────────────────────┐
│  1. NODE.md is authority       → controls the folder  │
│  2. Leaf owners are additive   → both can approve     │
│  3. Multiple owners OK         → [alice, bob]         │
│  4. Inheritance with override  → child overrides      │
│     (owners: [] = inherit)                            │
│  5. Every folder needs NODE.md → no sprawl            │
│  6. Wildcard [*]               → open + parent keeps  │
│                                  authority             │
└───────────────────────────────────────────────────────┘
```

```
┌───────────────────────────────────────────────────────┐
│  7. All owners must approve    → AND logic            │
│  8. Fallback = new PR          → clean git history    │
└───────────────────────────────────────────────────────┘

## CODEOWNERS generation

The CLI can generate a GitHub `CODEOWNERS` file from the tree's ownership declarations. This gives you PR-level enforcement for free.

```
# Auto-generated from Context Tree. Do not edit manually.
/backend/          @alice
/backend/auth.md   @alice @carol
/frontend/         @dana @eli
```

No custom tooling needed. The tree's ownership model maps directly to GitHub's native review system.
