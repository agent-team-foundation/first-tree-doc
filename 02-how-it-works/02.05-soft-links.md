# Soft Links

A tree has one rule: every node has exactly one parent. But sometimes two nodes in different parts of the tree are related.

Soft links solve this.

## The problem

```
/backend/auth.md        ← owns the auth design
/frontend/login.md      ← needs to know about auth

These are related, but they live in
different subtrees. The tree structure
does not connect them.
```

## The solution

Declare the relationship in frontmatter.

```yaml
# /frontend/login.md
---
title: Login Page
owners: [dana]
soft_links: [/backend/auth.md]
---
```

The soft link says: "this node is related to that node." An agent traversing the tree can follow soft links to build broader context when needed.

## How it works

```
                    root
                   /    \
             backend    frontend
              /            \
         auth.md        login.md
              │              │
              └──soft link───┘

The tree stays a tree.
The link is metadata, not structure.
```

Soft links are:

- **Declared in frontmatter.** Not in the file structure.
- **One-directional.** `login.md` links to `auth.md`. The reverse is not implied.
- **Optional.** Most nodes do not need them.
- **Non-structural.** They do not change the hierarchy. The tree remains a tree.

## When to use them

Use soft links when a node in one subtree needs context from a node in another subtree, and an agent reading the first node should be aware of the second.

Do not use soft links to replicate a graph. If two nodes are so tightly coupled that they always need each other, they probably belong in the same subtree.

## Tree, not graph

```
Tree + soft links:              Graph:

     root                       A ── B ── C
    /    \                      │ ╲  │ ╱  │
   A      B                    D ── E ── F
  / \    / \
 C   D  E   F                  Unnavigable.
      \   /                    Where do you start?
    soft link

Navigable. Clear hierarchy.
Cross-references where needed.
```

Soft links give you the 10% of graph functionality you actually need, without the 90% of complexity you do not.
